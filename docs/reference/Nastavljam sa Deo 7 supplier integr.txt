Nastavljam sa **Deo 7: supplier integration architecture, mapping, normalization i error handling**.
Ovo je deo koji je najbliži realnom radu platforme jer **bez stabilnih supplier integracija booking engine ne postoji**.

Ovaj deo objašnjava:

* kako se povezuju dobavljači (API, XML, GraphQL)
* kako se njihovi različiti podaci normalizuju
* kako se mapiraju hoteli i proizvodi
* kako se rešavaju greške
* kako se radi fallback između dobavljača
* kako se prati kvalitet integracije

---

# Unified Travel Booking Platform

## Deo 7 — Supplier Integration Architecture

---

# 1. Zašto je potreban integration layer

Svaki dobavljač ima različite:

* API strukture
* modele cena
* identifikatore hotela
* pravila rezervacije
* statuse
* payload formate

Primer:

Jedan hotel može doći iz:

* Hotelbeds
* Expedia
* direct hotel contract
* channel manager
* DMC
* wholesaler

Ako platforma direktno koristi supplier API, sistem postaje **neodrživ**.

Zato je potreban **normalization layer**.

---

# 2. Preporučena arhitektura integracija

```text
Frontend
   |
API Gateway
   |
Search Service
   |
Supplier Aggregation Layer
   |
Normalization Layer
   |
Supplier Connectors
   |
External Supplier APIs
```

---

# 3. Supplier connector sloj

Svaki dobavljač ima svoj **connector modul**.

Primeri:

```text
connector_hotelbeds
connector_expedia
connector_ratehawk
connector_duffel
connector_amadeus
connector_viator
connector_getyourguide
connector_msc_cruises
connector_direct_hotel
```

Svaki connector:

* zna supplier API
* radi autentifikaciju
* prevodi request
* prevodi response

---

# 4. Connector odgovornosti

Connector mora da radi:

1. autentifikaciju
2. request formatiranje
3. response parsing
4. error parsing
5. retry logiku
6. timeout handling
7. response caching
8. metrics logging

Ali **connector ne sme odlučivati poslovna pravila**.

To je posao višeg sloja.

---

# 5. Normalization layer

Ovo je najvažniji deo.

Različiti supplier odgovori moraju biti prevedeni u **standardni model platforme**.

Primer:

Supplier A:

```json
{
 "hotelCode": "HT123",
 "rate": 220,
 "currency": "EUR"
}
```

Supplier B:

```json
{
 "property_id": "9981",
 "price": {
   "amount": 215,
   "currency": "EUR"
 }
}
```

Normalization layer mora vratiti:

```json
{
 "hotelId": 1201,
 "supplierId": 5,
 "supplierHotelCode": "HT123",
 "roomType": "double",
 "netPrice": 220,
 "currency": "EUR"
}
```

---

# 6. Normalizovani modeli

Platforma mora imati **canonical modele** za:

* hotel
* hotel room
* hotel rate
* flight itinerary
* transfer offer
* activity offer
* cruise cabin offer

Ovo omogućava da ostatak sistema radi nezavisno od dobavljača.

# 13. Supplier mapping

Jedan hotel može imati **više supplier ID-jeva**.

Tabela mapiranja:

```text
hotel_id | supplier_id | supplier_hotel_code
```

Primer:

```text
1201 | hotelbeds | HT123
1201 | expedia | 9981
1201 | ratehawk | 55290
```

To omogućava:

* deduplikaciju rezultata
* bolji izbor cene
* fallback između dobavljača

---

# 14. Room mapping

Supplier room nazivi su često različiti.

Primer:

```text
Supplier A: Deluxe Double
Supplier B: Double Deluxe Room
Supplier C: DLX DBL
```

Sistem mora mapirati u interni model.

```text
room_type = deluxe_double
```

---

# 15. Board mapping

Supplier može koristiti:

```text

```

Sistem mora mapirati na standard:

```text

```

# 16. Cancellation policy mapping

Supplier može vratiti:

```text
free_cancel_until

partial_refund
```

Interni model:

```text
cancellation_type
free_until_date
penalty_percent
penalty_amount
```

---

# 17. Search orchestration

Kada korisnik pokrene pretragu, sistem može pozvati više supplier-a paralelno.

Primer:

```text
hotelbeds
ratehawk
expedia
direct_contract
```

Search service:

1. šalje request svim connector-ima
2. čeka odgovore
3. normalizuje podatke
4. deduplikuje hotele
5. prosleđuje package engine-u

---

# 18. Parallel request model

```text
Search request
   |
parallel requests
   |
supplier A
supplier B
supplier C
supplier D
   |
Normalization
   |
Aggregation
```

---

# 19. Timeout pravila

Supplier API mora imati timeout.

Primer:

```text
hotel search timeout = 4s
flight search timeout = 6s
activity search timeout = 3s
```

Ako supplier ne odgovori:

* sistem ne blokira rezultate
* samo preskoči tog supplier-a

---

# 20. Retry pravila

Retry je dozvoljen samo za:

* network timeout
* 5xx errors

Primer:

```text
retry = 2
retry_delay = 300ms
```

Ali ne za:

* invalid request
* no availability

---

# 21. Supplier fallback logika

Ako isti hotel postoji kod više dobavljača:

Sistem bira:

1. najnižu cenu
2. ako su iste — veću maržu
3. ako su iste — pouzdanijeg supplier-a

---

# 22. Supplier reliability score

Svaki supplier mora imati metriku pouzdanosti.

Primer:

```text
confirmation_rate
timeout_rate
error_rate
average_response_time
```

Iz toga se računa score.

---

# 23. Supplier monitoring

Sistem mora pratiti:

* broj API poziva
* broj grešaka
* timeout
* response time
* success booking rate

To ide u **supplier dashboard**.

---

# 24. Error kategorije

## Supplier errors

* no availability
* invalid rate
* rate expired
* booking rejected

## Technical errors

* timeout
* network error
* invalid response
* authentication error

---

# 25. Error handling model

```text
supplier request
→ response
→ parse
→ if error
     classify error
     log error
     retry if allowed
     return normalized error
```

---

# 26. Booking supplier flow

Kada booking ide ka supplier-u:

```text
reserve
→ confirm
→ return supplier reference
```

Supplier reference se čuva u:

```text
supplier_reservations table
```

---

# 27. Supplier sandbox / test

```text
sandbox

production
```

Test scenariji moraju uključiti:

* successful booking

* price change

# 34. Fallback između dobavljača

Ako booking kod supplier A padne:

Sistem može pokušati supplier B za isti hotel.

Ali samo ako:

* rate je kompatibilan
* cena nije drastično drugačija
* korisnik prihvati novu cenu
