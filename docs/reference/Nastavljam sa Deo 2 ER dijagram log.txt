# TEHNIČKI DOKUMENT ZA RAZVOJNI TIM

## Unified Travel Booking Platform

### Deo 2 — ER dijagram logika + detaljna SQL schema

# 1. Arhitektonski princip baze

Baza je organizovana kroz 4 glavne domene:

1. **Identity & Access**
2. **Product & Content**
3. **Search / Offer / Booking**
4. **Finance / Documents / Admin**

Suština modela je:

* jedan **master booking**
* više **booking items**
* svi proizvodi tretirani kao zasebne stavke
* offer/snapshot sloj između pretrage i bookinga
* jasna odvojenost između:

  * statičkog sadržaja
  * dinamičke raspoloživosti i cena
  * transakcionih zapisa

# 3. Identity & Access schema

## 3.1 users

```sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash TEXT,
    first_name VARCHAR(120),
    last_name VARCHAR(120),
    phone VARCHAR(60),
    country_code VARCHAR(10),
    locale VARCHAR(10),
    status VARCHAR(30) NOT NULL DEFAULT 'active',
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```

## 3.2 roles

```sql
CREATE TABLE roles (
    id BIGSERIAL PRIMARY KEY,
```

# 17. Minimalni ER zaključci po ključnim relacijama

## Jedan korisnik

* ima više putnika
* može imati više rezervacija
* može biti i agent ako ima agent profile

## Jedna rezervacija

* ima više booking items
* ima više putnika
* može imati više payment zapisa
* može imati više dokumenata
* nastaje iz jednog offer-a i jednog pricing snapshot-a

## Jedan booking item

* pripada jednoj rezervaciji
* ima jedan supplier
* može imati više supplier reservation logova
* može imati sopstveni dokument
* može biti povezan sa više putnika

## Jedan offer

* pripada jednoj search request instanci
* ima više offer items
* ima jedan ili više pricing snapshot-a
* može kasnije postati booking

# 18. Zašto je ovaj model dobar

Ovaj model rešava nekoliko ključnih problema:

## 1. Multi-product booking

Jedna rezervacija može imati više različitih proizvoda.

## 2. Supplier heterogenost

Svaka stavka čuva supplier reference odvojeno.

## 3. B2B i B2C u istoj bazi

Kanal prodaje je deo svakog booking-a i offer-a.

## 4. Paketiranje

Offer i offer_items služe kao most između search i booking faze.

## 5. Post-booking fleksibilnost

Moguće je otkazivati pojedine stavke i pratiti njihov status.

---

# 19. Šta bih preporučio razvojnom timu odmah posle ovog dela

Posle ove baze treba zaključati još 3 stvari:

1. **enum vrednosti i status dictionary**
2. **index strategiju**
3. **pravila za offer expiration i repricing**

To je važno pre izrade pravog SQL deployment-a.
