Odlično. Krećem sada na **Detailed Data Model** — onako kako to rade CTO i product arhitekte kada platforma pređe iz faze ideje u fazu **realne implementacije**.

Ovde neću raditi “grub pregled”, nego ću postaviti:

* **kompletnu logiku baze**
* **raspodelu na domene**
* **strukturu od ~500+ tabela**
* **ključne relacije**
* **API boundary**
* **event / queue model**
* **pravila imenovanja i verzionisanja**

Ovo je praktično **Master Data Architecture V1**.

---

# 3. Naming standard

Da baza ostane održiva, imena moraju biti strogo standardizovana.

## 3.1 Pravila

* sve tablice u `snake_case`
* svi PK: `id`
* svi FK: `<entity>_id`
* enum kolone: `<name>_code`
* date/datetime:

  * `created_at`
  * `updated_at`
  * `deleted_at`
* status kolone:

  * `status_code`
* iznosi:

  * `amount`
  * `currency_code`
* audit:

  * `created_by_user_id`
  * `updated_by_user_id`

## 3.2 Primer

* `bookings`
* `booking_items`
* `booking_item_passengers`
* `hotel_contracts`
* `hotel_contract_rate_seasons`
* `charter_flight_series`
* `mail_threads`
* `campaign_audience_members`

---

# 4. Multi-tenant / brand-ready logika

Pošto imate:

* B2C sajt
* B2B mrežu
* moguće više brendova
* future white-label

predlažem da ključne tabele imaju:

* `brand_id`
* `market_id`
* `channel_id`

gde je smisleno.

To ne mora svuda, ali mora postojati kao mogućnost.

---

# 5. ID strategija

Za enterprise sistem preporučujem:

* interni PK = `bigint`
* public ref = human-friendly code
* external ref = supplier reference

Primer:

* `bookings.id = 1288821`
* `bookings.booking_reference = OT-2026-00482`
* `supplier_reservations.external_reference = HB-884221`
