Odlično. Nastavljam sa sledećim velikim slojem:

# DB DEPLOYMENT STRATEGIJA

## Data layer, schemas, partitioning, indexing, read/write, reporting, archive, backup

Ovo je deo koji određuje da li će sistem:

* ostati brz kad poraste broj rezervacija
* moći da podnese search load
* izdržati veliki broj mailova, kampanja i AI logova
* imati čiste reporting i operativne upite
* biti održiv 3–5 godina bez haosa

Ovde definišemo:

* organizaciju baze po šemama
* strategiju read/write razdvajanja
* particionisanje
* indeksiranje
* materialized views
* reporting sloj
* arhiviranje
* backup / recovery
* data retention

---

# 1. OSNOVNI PRINCIP

Za ovakav sistem preporučujem:

## 1.1 Primarni OLTP sloj

**PostgreSQL** kao glavna transakciona baza.

Razlog:

* odličan za booking/finance/CRM
* jak ACID model
* odličan za JSONB gde treba
* podržava particionisanje
* odličan za audit i relacione podatke

## 1.2 Pomoćni slojevi

Pored primarne baze:

* **Redis** za cache/session/temporary locks
* **Elasticsearch / OpenSearch** za full-text i search index
* **Object Storage** za dokumenta i attachment-e
* **Data Warehouse** za tešku analitiku
* **Queue/Event storage** kroz DB + broker kombinaciju

Dakle:

```text
PostgreSQL = system of record
Redis = speed layer
Search index = retrieval layer
Object storage = files
Warehouse = analytics layer
```

* `(related_agency_id)`

## `charter.charter_flights`

* `(charter_flight_series_id, flight_date, direction_code)` unique
* `(flight_date, status_code)`
* `(status_code, seat_capacity_free)`

# 21. SOFT DELETE VS HARD DELETE

## Soft delete koristiti za:

* contacts
* agencies
* products
* pages
* templates
* bookings samo izuzetno oprezno

## Hard delete koristiti za:

* cache entries
* transient queue jobs
* old temporary snapshots
* test data
* some imports after retention

Za većinu core business entiteta:
**soft delete + audit** je bolji.

* **RTO < 1 h**

## Mail / communication

* RPO nizak, ali može nešto viši od payment layer-a ako postoji replay

## Marketing / analytics

* viši tolerisani RPO/RTO

# 24. DISASTER RECOVERY

Mora postojati plan za:

* primary DB failure
* region outage
* broker/queue outage
* mail provider outage
* search cluster outage

## Minimalno:

* replica / failover
* restore tested
* runbook dokumentovan
* queue replay podržan
* degraded mode za website/public search
