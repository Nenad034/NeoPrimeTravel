Nastavljam sa **Deo 8: infrastruktura sistema, microservice arhitektura, skaliranje i deployment model**.
Ovo je deo koji objašnjava **kako platforma fizički i tehnički funkcioniše u produkciji**, posebno kada ima veliki broj pretraga i rezervacija — što je tipično za travel industriju.

# Unified Travel Booking Platform

## Deo 8 — Infrastructure, Microservices i Skaliranje

---

# 1. Glavni ciljevi infrastrukture

Travel platforma mora da bude:

* **visoko dostupna**
* **skalabilna**
* **otporna na greške**
* **brza u pretrazi**
* **stabilna u rezervacijama**

Posebno jer travel sistemi imaju:

* ogromne search volumene
* relativno mali procenat rezervacija
* mnogo spoljašnjih API zavisnosti

Tipičan odnos:

```text
1000 search request
→ 100 offer views
→ 10 booking attempts
→ 3–5 confirmed bookings
```

# 3. Frontend sloj

* React / Next.js
* Vue
* Angular

Preporuka za modernu travel platformu:

**Next.js + React**

Razlog:

* API kompatibilnost

Frontend komunicira sa backendom preko:

```text
API Gateway
```

Funkcije:

* autentifikacija
* rate limiting
* routing

* request validation

* Kong

# 5. Microservice arhitektura

Platforma treba da bude podeljena u više servisa.

Primer:

```text
search-service
hotel-service
flight-service
package-service
transfer-service
activity-service
cruise-service
pricing-service
booking-service
payment-service
supplier-service
notification-service
reporting-service
```

Svaki servis ima svoju odgovornost.

---

# 6. Search service

Search service je najopterećeniji deo sistema.

Odgovoran za:

* pretragu
* agregaciju supplier rezultata
* deduplikaciju
* ranking
* filtriranje
* pagination

Search mora biti **visoko optimizovan**.

---

# 7. Booking service

Booking service je **kritičan servis**.

Odgovoran za:

* kreiranje rezervacije
* orchestration
* status promene
* supplier rezervacije
* rollback
* dokumenta

Za razliku od search servisa, mora biti:

* konzistentan
* pouzdan
* auditabilan

---

# 8. Pricing service

Pricing service mora biti izdvojen.

Razlog:

* svi proizvodi koriste pricing
* pravila se često menjaju
* admin panel upravlja pravilima

Pricing service radi:

* markup
* commission
* bundle discount
* promo
* final price

---

# 9. Supplier integration service

Ovaj servis komunicira sa svim dobavljačima.

Odgovoran za:

* supplier API calls
* normalization
* retry
* timeout handling
* caching

Ovaj servis mora imati **rate limiting**.

---

# 10. Notification service

Notification service šalje:

* email
* SMS
* push notification
* internal alerts

Primeri:

* booking confirmation
* payment confirmation
* cancellation
* supplier update

# 11. Message queue

Message queue je ključan za skalabilnost.

Primer:

```text

* RabbitMQ

* AWS SQS
* Google Pub/Sub

# 12. Worker servisi

Neki procesi ne treba da blokiraju API.

Primer:

* document generation
* email sending
* supplier retries
* analytics logging

Worker servisi obrađuju te zadatke.

# 13. Database arhitektura

 |
Read replicas
 |
Analytics database
```

PostgreSQL je odličan izbor za travel platformu.

---

# 14. Baza po servisu vs monolitna baza

Dve strategije postoje.

## Monolitna baza

Jedna baza za sve servise.

Prednost:

* lakše održavanje

Mana:

* teže skaliranje

## Baza po servisu

Svaki servis ima svoju bazu.

Prednost:

* skalabilnost

Mana:

* kompleksnije održavanje

Za MVP je prihvatljivo početi sa **jednom bazom**.

---

# 15. Cache layer

Cache je kritičan za search performanse.

Tehnologije:

* Redis
* Memcached

Cache koristi se za:

* search results
* hotel content
* destination data
* supplier responses

---

# 16. Search caching model

Primer:

```text
User search
→ check cache
→ if cached return results
→ else query suppliers
→ store in cache
```

TTL može biti:

```text
search cache = 2–5 min
```
