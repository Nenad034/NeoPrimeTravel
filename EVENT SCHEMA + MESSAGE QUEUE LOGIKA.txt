# 1. Osnovni princip

Ne treba sve raditi event-driven.
Treba razdvojiti:

## 1.1 Sync-critical tokove

moraju dati odgovor odmah

Primer:

* login
* search response
* offer reprice
* booking create request accepted
* payment intent create

## 1.2 Async-operational tokove

mogu i treba da idu preko queue/event sistema

Primer:

* dokument generacija
* slanje email-a
* rooming export
* manifest export
* campaign delivery
* AI scoring
* publishing refresh

# 2. Event arhitektura — osnovni model

Predlažem sledeću arhitekturu:

```text
Domain Service
   ↓
Outbox Table
   ↓
Event Publisher
   ↓
Event Bus / Queue
   ↓
Subscribers / Workers
   ↓
Read models / Side effects / Notifications
```

To znači:

* domen upiše svoju promenu u bazu
* zajedno sa outbox zapisom
* publisher šalje event
* worker-i rade dalje

To je najzdraviji pristup.

---

# 3. Zašto outbox pattern

Za booking i finansije je veoma važno da ne dođe do situacije:

* booking je upisan
* ali event nije poslat

ili obrnuto:

* event poslat
* booking nije upisan

Zato preporučujem:

## Transactional Outbox Pattern

Svaki domen koji emituje važne evente upisuje u istoj DB transakciji:

* business record
* outbox record

---

# 4. Event naming standard

Event imena moraju biti strogo standardizovana.

Predlažem format:

```text
<domain>.<entity>.<action>
```

Primeri:

* booking.booking.created
* booking.item.confirmed
* finance.payment.received
* rooming.set.sent
* mail.thread.assigned
* marketing.campaign.sent
* website.page.published

Ako je potrebno:

```text
<domain>.<entity>.<subentity>.<action>
```

Primer:

* charter.flight.load_factor.updated

# 7. Correlation i causation

# 27. RETRY PRAVILA

Retry ne sme biti isti za sve.

## 27.1 Retry policy tipovi

### A. Network / transient retry

Za:

* supplier API timeout
* mail provider timeout
* temporary DB lock
* external webhook down

Predlog:

* 3 do 5 pokušaja
* exponential backoff

### B. Business error — no retry

Za:

* invalid request
* stop sale
* insufficient credit
* invalid passenger data

### C. Conditional retry

Za:

* supplier pending state
* async ticketing
* delayed confirmation

---

# 28. Exponential backoff predlog

Na primer:

* attempt 1 → odmah
* attempt 2 → 30 sec
* attempt 3 → 2 min
* attempt 4 → 10 min
* attempt 5 → 30 min

Za neke marketing i AI taskove može i sporije.

# 29. DEAD LETTER QUEUE

Kada worker ne može da obradi poruku posle max retry:

* ide u DLQ

## DLQ svrha

* ništa se ne gubi
* support/ops/tech može ručno reagovati
* postoji forenzika

## `dead_letter_jobs`

ili `queue_dead_letters`
već smo definisali u modelu.

### Polja koja su obavezna

* original job id
* payload
* error history
* final failure reason
* dead_lettered_at

# 41. FAILURE HANDLING PO DOMENIMA

## Booking

* fail fast
* alert ops
* create incident
* preserve partial state

## Finance

* never lose payment signal
* queue replay mandatory
* manual review queue

## Mail

* retry provider send
* never duplicate send
* if fail → visible send failure state

## Marketing

* partial batch retry allowed
* recipient-level tracking

## AI

* fallback model or no-ai response
* never block mission-critical booking flow
