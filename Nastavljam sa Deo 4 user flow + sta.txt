# TEHNIČKI DOKUMENT ZA RAZVOJNI TIM

## Unified Travel Booking Platform

### Deo 4 — User flow, state machine i booking orchestration

# 1. Svrha ovog dela

Ovaj deo definiše:

1. **korisničke tokove po proizvodu**
2. **state machine za offer, booking i booking item**
3. **booking orchestration**
4. **rollback logiku**
5. **partial confirmation scenarije**
6. **post-booking tokove**

Cilj je da svi timovi imaju isti model ponašanja sistema.

# 2. Glavni koncepti toka

Sistem ima 4 ključne faze:

```text
SEARCH
→ OFFER
→ REPRICE
→ BOOKING
```

Posle toga ide:

```text
POST-BOOKING SERVICING
```

Važno pravilo:

* **Search** služi za prikaz mogućnosti
* **Offer** je konkretna kombinacija spremna za izbor
* **Reprice** proverava da li je sve i dalje validno
* **Booking** kreira transakciju i rezerviše kod dobavljača

# 5. OFFER STATE MACHINE

Offer predstavlja opciju koju korisnik može da izabere.

## 5.1 Offer statusi

* `available`
* `expired`
* `repricing_required`
* `repriced`
* `unavailable`
* `booked`

## 5.2 Prelazi stanja

```text
available
→ repricing_required
→ repriced
→ booked
```

ili

```text
available
→ expired
```

ili

```text
available
→ unavailable
```

## 5.3 Pravila

### `available`

Offer postoji i može da se prikaže korisniku.

### `repricing_required`

Koristi se kada:

* je flight/package volatile
* je istekao stari price snapshot
* je potreban novi check dostupnosti

### `repriced`

Offer je ponovo potvrđen i može u booking flow.

### `expired`

Vreme važenja je isteklo.

### `unavailable`

Jedna ili više komponenti više ne postoji.

### `booked`

Offer je već iskorišćen za kreiranje rezervacije.

# 6. BOOKING STATE MACHINE

Booking je master entitet.

## 6.1 Booking statusi

* `pending`
* `awaiting_payment`
* `payment_in_progress`
* `processing`
* `confirmed`
* `partially_confirmed`
* `failed`
* `cancel_requested`
* `cancelled`
* `refunded`

## 6.2 Tok

### Standardni uspešan tok

```text
pending
→ awaiting_payment
→ payment_in_progress
→ processing
→ confirmed
```

### Ako deo stavki uspe, a deo ne uspe

```text
processing
→ partially_confirmed
```

### Ako sve padne

```text
processing
→ failed
```

### Otkazivanje

```text
confirmed
→ cancel_requested
→ cancelled
```

### Refund

```text
cancelled
→ refunded
```

## 6.3 Značenje statusa

### `pending`

Booking zapis je kreiran, ali dalje radnje još nisu počele.

### `awaiting_payment`

Čeka se iniciranje ili potvrda plaćanja.

### `payment_in_progress`

Payment gateway je aktivan.

### `processing`

Backend rezerviše jednu po jednu stavku.

### `confirmed`

Sve stavke potvrđene.

### `partially_confirmed`

Neke stavke potvrđene, neke nisu.

### `failed`

Rezervacija nije uspela.

### `cancel_requested`

Korisnik ili agent je zatražio otkazivanje.

### `cancelled`

Otkazivanje je obrađeno.

### `refunded`

Refund je izvršen ili evidentiran.

# 9. PREPORUČENI REDOSLED OBRADE

Redosled nije isti za sve proizvode, ali predlažem sledeću logiku.

## 9.1 Hotel only

```text
1. revalidate hotel item
2. authorize payment
3. reserve hotel
4. capture payment
5. generate voucher
6. confirm booking
```

## 9.2 Flight only

```text
1. revalidate flight fare
2. authorize payment
3. reserve flight
4. ticket if needed
5. capture payment
6. generate itinerary
7. confirm booking
```

## 9.3 Hotel + flight

```text
1. revalidate flight
2. revalidate hotel
3. authorize payment
4. reserve flight
5. reserve hotel
6. if both success → capture payment
7. generate documents
8. confirm booking
```

## 9.4 Hotel + flight + transfer + activity

```text
1. revalidate all items
2. authorize payment
3. reserve flight
4. reserve hotel
5. reserve transfer
6. reserve activity
7. if all success → capture payment
8. generate documents
9. confirm booking
```

## 9.5 Cruise package

```text
1. revalidate cruise
2. revalidate flight/hotel/transfer if any
3. authorize payment
4. reserve flight
5. reserve hotel if pre/post stay exists
6. reserve transfer
7. reserve cruise
8. capture payment
9. generate documents
10. confirm booking
```

# 10. ZAŠTO JE REDOSLED BITAN

Redosled mora biti poslovno smislen.

## 10.1 Flight pre hotela

Često ima smisla prvo rezervisati let jer:

* flight fare može brže da nestane
* često je volatilniji od hotela

## 10.2 Hotel pre transfera

Transfer zavisi od leta i/ili hotela.

## 10.3 Activity posle core komponenti

Izlet ili ulaznica su obično sekundarna komponenta.

## 10.4 Cruise kao anchor proizvod

Kod cruise package-a često je cruise glavna komponenta, ali operativno može biti pametno proveriti flight/hotel kompatibilnost unapred.

# 11. PAYMENT ORCHESTRATION

## 11.1 Preporučeni model

Za multi-item booking:

* prvo **authorize**
* zatim rezerviši stavke
* onda **capture**

## 11.2 Tok

```text
create booking
→ authorize payment
→ reserve items
→ if success capture payment
→ else void/cancel authorization
```

## 11.3 Ako authorize ne uspe

* booking ostaje `awaiting_payment` ili `failed`
* supplier booking se ne pokreće

## 11.4 Ako authorize uspe, a booking padne

* authorization se void/rollback
* booking ide u `failed`

---

# 12. ROLLBACK LOGIKA

Rollback je ključan za pakete.

## 12.1 Potpuni rollback

Koristi se kada želite “all-or-nothing” model.

Primer:

* flight uspe
* hotel padne

Akcija:

* otkaži flight hold / reservation ako je moguće
* ne potvrđuj booking
* vrati payment authorization

Rezultat:

* booking = `failed`

---

## 12.2 Delimični rollback

Koristi se kada je dozvoljeno da booking opstane bez sekundarne komponente.

Primer:

* flight uspe
* hotel uspe
* transfer padne

Moguća pravila:

* ostavi booking bez transfera
* booking = `partially_confirmed`
* korisniku se prikaže da transfer nije potvrđen

Ovo treba definisati po poslovnim pravilima.

---

# 13. PREPORUČENI ROLLBACK REŽIMI

## 13.1 Core items

Za sledeće stavke preporučujem stroži režim:

* flight

Ako padne core item, najčešće:

* ceo booking propada

## 13.2 Secondary items

Za sledeće stavke može postojati fleksibilnost:

* transfer
* activity
* insurance
* ticket add-on

Ako padne secondary item:

* booking može ostati `partially_confirmed`

# 14. PARTIAL CONFIRMATION SCENARIJI

## 14.1 Primer 1

Flight + hotel + transfer

* flight confirmed
* hotel confirmed
* transfer failed

Rezultat:

* booking = `partially_confirmed`
* documents za flight/hotel generisani
* transfer nije uključen
* korisnik dobija obaveštenje

## 14.2 Primer 2

Flight + hotel + activity

* flight confirmed
* hotel failed
* activity not attempted or cancelled

Rezultat:

* rollback flight ako je moguće
* booking = `failed`

## 14.3 Primer 3

Cruise + hotel pre-stay + transfer

* cruise confirmed
* hotel confirmed
* transfer failed

Rezultat:

* booking može biti `partially_confirmed`

flight = core

# 16. ORCHESTRATION PSEUDOKOD

## 16.1 Standardni multi-item booking

```python
def process_booking(booking):
    set_booking_status(booking, "processing")

    authorize_payment(booking)

    for item in ordered_items(booking):
        revalidate_item(item)

    for item in ordered_items(booking):
        result = reserve_item(item)

        if not result.success:
            handle_item_failure(booking, item, result)
            return

    capture_payment(booking)
    generate_documents(booking)
    set_booking_status(booking, "confirmed")
```

---

## 16.2 Failure handler

```python
def handle_item_failure(booking, failed_item, result):
    if is_core_item(failed_item):
        rollback_confirmed_items(booking)
        void_payment(booking)
        set_booking_status(booking, "failed")
    else:
        mark_item_failed(failed_item)
        finalize_partial_booking(booking)
        set_booking_status(booking, "partially_confirmed")
```

# 28. SEKVENCA ZA PACKAGE BOOKING

Primer: hotel + flight + transfer + activity

```text
1. User selects package offer
2. System reprices package
3. User enters travelers
4. Booking created as pending
5. Payment authorized
6. Booking status → processing
7. Flight item reserved
8. Hotel item reserved
9. Transfer item reserved
10. Activity item reserved
11. Payment captured
12. Documents generated
13. Booking → confirmed
```

Ako activity padne:

```text
11. Activity item failed
12. If optional → booking partially_confirmed
13. Capture adjusted payment or create refund delta rule
14. Documents for successful items generated
```
