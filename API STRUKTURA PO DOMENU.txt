Odlično. Nastavljam sa sledećim logičnim slojem:

# API STRUKTURA PO DOMENU

## Kompletna tehnička specifikacija endpoint-a

Ovo je deo koji razvojni tim koristi da razume:

* kako frontend komunicira sa sistemom
* kako B2C sajt komunicira sa platformom
* kako B2B portal komunicira sa platformom
* kako admin/backoffice radi
* kako eksterne integracije ulaze u sistem
* kako AI i event sloj koriste API granice

Neću ići u “samo primere”, nego u **realnu enterprise API arhitekturu**.

---

# 1. API arhitektura — osnovni princip

Predlažem da platforma ima **5 glavnih API slojeva**:

1. **Public Website API**
2. **B2C Customer API**
3. **B2B Partner API**
4. **Internal Admin / Operations API**
5. **Integration & System API**

To znači da ne postoji “jedan ogroman API”, nego jasno odvojeni konteksti.

---

# 2. Global API principi

## 2.1 Verzija

Svi endpoint-i moraju biti verzionisani.

Primer:

```text
/api/v1/...
```

Kasnije:

```text
/api/v2/...
```

## 2.2 Format

* JSON kao default
* multipart za upload
* CSV/Excel/PDF export preko posebnih download endpoint-a

## 2.3 Identifikatori

Svaki entitet ima:

* interni `id`
* javni `reference` ili `code`

Frontend i integracije treba da koriste:

* `reference`
* `code`
  gde god je moguće

---

# 3. Standard response model

Preporučujem standardizovan odgovor.

## 3.1 Success response

```json
{
  "success": true,
  "data": {},
  "meta": {},
  "errors": []
}
```

## 3.2 Error response

```json
{
  "success": false,
  "data": null,
  "meta": {},
  "errors": [
    {
      "code": "BOOKING_NOT_FOUND",
      "message": "Booking not found."
    }
  ]
}
```

## 3.3 Pagination

```json
{
  "success": true,
  "data": [],
  "meta": {
    "page": 1,
    "per_page": 20,
    "total": 140,
    "total_pages": 7
  },
  "errors": []
}
```

---

# 4. Auth model po API sloju

## Public Website API

* bez login-a za public content/search
* optional session token za personalization

## B2C API

* customer auth token
* refresh token
* session management

## B2B API

* agency auth token
* user scope + agency scope
* optional API key za B2B API access

## Internal API

* SSO/JWT/internal session
* role + permission enforcement

## Integration API

* API keys
* OAuth gde treba
* signed webhook validation

# 11. API DESIGN PRAVILA

## 11.1 Filtering

Svi list endpoint-i moraju podržati:

* pagination
* sorting
* basic filters
* date ranges
* status filters

Primer:

```text
GET /api/v1/admin/bookings?status=confirmed&travel_start_from=2026-07-01&travel_start_to=2026-07-31&page=1&per_page=50
```

## 11.2 Bulk endpoints

Za operativu su obavezni bulk endpoint-i:

```text
POST /api/v1/admin/bookings/bulk-assign
POST /api/v1/admin/operations/arrivals/bulk-export
POST /api/v1/admin/mail/threads/bulk-assign
POST /api/v1/admin/marketing/campaigns/{id}/bulk-preview
```

## 11.3 Idempotency

Za critical POST akcije:

* booking create
* payment create
* refund create
* supplier booking
  treba koristiti:
* `Idempotency-Key` header

# 12. API SECURITY PRAVILA

* JWT za B2C/B2B/internal
* API key + HMAC za system integrations gde treba
* rate limits po client-u i scope-u
* permission enforcement na internal API
* field-level masking za osetljive podatke
* audit log za write operacije

# 14. Sledeći korak

Sledeći prirodan korak je:

# EVENT SCHEMA + MESSAGE QUEUE LOGIKA

Tu ću detaljno definisati:

* event nazive
* event payload standard
* koji modul emituje koji event
* koji worker sluša koji event
* retry logiku
* dead letter pristup
* sinhrone vs asinhrone operacije

To je sledeći ključni deo koji pravi sistem **zaista spremnim za razvoj**.
